FieldVision: Planning Admin Tool

Introduction

Product Overview

FieldVision is an internal enterprise application used by 1800+ field engineers to manage the tasks required to service a power plant.

Prior to service, field engineers perform a set of planning activities, as outlined in FieldVision, to ensure service goes smoothly.

Every quarter, key business stakeholders review the planning activities and adjust them according to proposed changes sent to them.

Problem Statement

After finalizing the changes, our developers required 1 month of their limited bandwidth to properly implement them.

In that time, users either wait for FieldVision to update, or receive the updates manually and have to remember to mark activities in FieldVision as complete post-update.

Goal

I was challenged to design a Planning Admin Tool that would allow key stakeholders to directly edit planning activity data without the need of developers.

My Role

Sole UX Designer on feature

  • user research
  • competitive analysis
  • wireframes and prototyping
  • user testing
  • dev-ready documentation
  • constant collaboration with dev and product teams

Timeline

4 sprints (~2 months) before Dev team was scheduled to start building.

Challenges and Limitation

  • Limited bandwidth and tight turnaround time.
  • Group “demo sessions” done in place of 1-on-1 usability testing to save time.

Discovery

Understanding the Current Process

Before designing, I met with 4 key stakeholders to go over the existing process and the problems they encountered with it.

Key findings included:

  • Changes to activities were made by editing specific fields outlined in a “Master Planning Sheet” Excel file.
  • Keeping track of proposed changes was very manual, with stakeholders communicating through edits of Excel files sent back and forth over multiple email threads.

User Needs

Based on this user session, I determined the following key user needs:

  • Simple revisions - Whatever solution I created needed to be as simple as editing an Excel file, otherwise users would revert to previous Excel methods.
  • Visibility on revisions - Keeping changes in one centralized area would cut down on time navigating through different Excel sheets and email threads.

Competitive Analysis

The most common revision control tool I'd used was GitHub, so I thought if GitHub had figured out group revision control for a complex codebase, I could leverage their solutions for our planning activities.

My key takeaways from GitHub were:

  • GitHub allowed users to attribute each change with a “Commit message” for easier identification.
  • When users wanted to push changes, they would issue a “Pull Request” which let others review and approve their changes and/or leave comments for everyone else to see.

Design Process

Sketches

Even though our product team and stakeholders only responded to high fidelity wireframes, I find starting with sketches is the best way for me to generate ideas, uncover requirements, and make sure I fully understand the problem.

Through sketches, I distilled key actions, noted existing solutions we could leverage, and discovered I didn't fully understand how changes affected ongoing services or how equipment codes were determined.

Leveraging Existing Solutions

Data tables and forms already existed in the application, so rather than create new ones, I took those components and adjusted them for the Planning Admin Tool.

By reusing existing components, I saved time creating a whole new solution and vetting its feasibility with the dev team.

Breaking Down Competitive Analysis

I took each aspect of GitHub's solutions and broke them down into their core pieces. I then replaced these pieces with whatever was required for the Planning Admin Tool.

Feedback

Improving the Feedback Process

When I demoed my first iteration for feedback, I found two flaws in a group demo format:

  • Because I was giving a demo instead of users exploring the solution, I wasn't able to truly test my solution's usability.
  • Sometimes one loud voice dominates over the rest. It wasn't until the end of the demo that one stakeholder was able to tell me that about a previously unknown Steam Power use case.

I spoke with my Product Manager and made the following changes to future feedback sessions:

  • I shifted the format to be closer to usability tests — No more demos from me; users would be the ones driving the call.
  • Demos needed to be split between Gas Power and Steam Power stakeholders so we could collect role-specific feedback.

Usability Testing Results

Users pointed out that certain fields should be dropdowns or not editable at all, and also felt the arrangement of the form's fields didn't match the proper hierarchy. So I fixed the incorrect fields and rearranged the form's layout based on their feedback.

On the Steam form, I had checkboxes for fields that were Yes/No values. Due to their proximity to dropdowns, users thought the checkboxes were used to activate those dropdowns. I changed these checkboxes to dropdowns, which users felt made more sense.

Features and Designs

Planning Activity List Table

The main challenge with the table was fitting 20+ columns of Excel data into a table meant to fit on one screen. Additionally, while similar, the Gas and Steam Power attributes weren't quite one to one, so we needed two arrangements.

So for each business, I grouped previously individual attributes into single columns displaying multiple values. To maintain some of Excel's flexibility, users could filter the table and/or change the way activities were grouped.

Edit Activity Forms

From my competitive analysis, I took GitHub's commit view and replaced the file edit section with the Edit Activity Form.
For the form itself, I arranged the fields based on what I learned in my user feedback sessions.

In addition to a suggestion title and comment, a calendar was added so users could set when changes should go into effect.

What Didn't Make It

Below are some preliminary designs for features that weren't completed due to timing.

Stakeholders informed me equipment selection wasn't as straightforward as selecting from a list. However, they also told me that changing equipment was rare, so due to its complexity and low use we decided this could be a future enhancement.

I was also not able to design every aspect of version control in the given time. As valuable as version control would be to users' long-term needs, delivering the tool on time and giving stakeholders the ability to edit data was more important.

As such, full version control was moved to a future enhancement.

Outcomes

Results

After development of MVP1, the tool reduced the turnaround time for quarterly changes from 1 month to almost immediate.

Development on MVP2 was passed onto another UX Designer as I shifted me to another product

Reflections

While user research can be more difficult to accomplish in enterprise settings, it still needs to be done, in whatever capacity.

I learned how valuable it was to leverage existing solutions to work quickly and gather proper user feedback.

It's also important to understand what truly needs to be delivered and what can be cut, especially in agile settings.
